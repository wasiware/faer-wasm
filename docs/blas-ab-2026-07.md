# BLAS-layer A/B: streaming loops vs faer, per operation (2026-07-13)

Architect-directed hypothesis test ("wouldn't faer-wasm perform better if
everything was shaped?"): for every BLAS-level operation faer-wasm uses
unchanged from faer, a streaming-loop variant built on our SIMD
primitives, timed interleaved against the faer path on one machine —
5 alternating rounds, median [min..max], verdicts only on range
separation. Harness: `bench/blas-ab.mjs` + `run_blas_ab` (both variants
cross-checked for numerical agreement before timing). ratio > 1 means
the streaming loop is FASTER than faer.

**Environment caveat: this table is from the dev container** (a
desktop-class CPU with high memory bandwidth). Per the verdict-stability
rule, sub-1.3× margins are machine-weather — runner replication is the
open item before any of this graduates past "observed on one machine".

## Reading

1. **Level 1/2 parity confirmed** (copy, gemv, ger, trsv): 17 of 20 rows
   are OVERLAP, and no faer win separates anywhere. These operations are
   memory-bound — a shaped loop and faer's path finish in a dead heat
   because both are waiting on the same bytes. The standing policy
   (don't shape what bandwidth already caps) holds.
2. **The surprise is Level 3.** The working number for R — gemm
   throughput over streaming-loop throughput, the quantity the whole
   shaping strategy is calibrated against — has been ~2 on the runner.
   On THIS machine, measured directly, the flat SIMD column-axpy gemm
   comes within 4–10% of faer's blocked gemm at every size (loop/faer
   0.90–0.96, never separated). syrk reads at loop-parity-or-better
   (1.00–1.17, never separated; the triangular-aware loop skips half
   the writes). Only trsm (0.83–0.87 at n≥512) and trmm at 1024 (0.94)
   produce separated faer wins.
3. **Policy consequence: none yet — the verdicts do not justify
   replacing anything.** faer's paths never lose separated; the loop's
   apparent edges never separate. The unchanged rows stay unchanged.
4. **Thesis consequence if the runner replicates this: R is
   machine-dependent and possibly much closer to 1 than 2.** That would
   sharpen, not weaken, the common-thread story: (a) the case for
   deleting mid-layer blocked machinery gets stronger (even less gemm
   advantage to buy back a surcharge σ); (b) faer's 5–20× matmul wins
   over Pyodide are then about scipy's OpenBLAS-generic gemm being far
   from the machine ceiling, not about blocking beating streaming;
   (c) the multishift's large-n win over our hand kernel must be
   credited to aggressive early deflation (fewer sweeps — an
   algorithmic effect), not gemm batching. Each of those is checkable.

## Runner replication — three draws (2026-07-18, runs 29631062796 / 29631536777 / 29631542527)

The reference-environment answer, and it inverts the container's L3
reading. Consensus across all three runner draws:

- **n ≤ 512, Level 3: the streaming loop BEATS faer's blocked path on
  every draw, 1.1–1.45×, mostly range-separated.** gemm itself: 1.17×/
  1.21×/1.18×/1.21× (draw 1, all separated), 1.31×/1.07×/1.15×/1.14×
  (draw 2), 1.33×/1.18×/1.17×/1.22× (draw 3) at n=64–512. syrk, trmm,
  trsm read the same way.
- **n = 1024 is machine-dependent**: draws 1 and 3 stay loop-ahead
  (gemm 1.21×/1.22× separated); draw 2 reads parity-to-faer (gemm
  1.01× OVERLAP, trmm 0.91× faer WINS) — a cache-size effect,
  consistent with the verdict-stability rule.
- Level 2: loop modestly ahead everywhere (1.05–1.25×, frequently
  separated). Level 1 copy: parity.

**Directly measured on the reference machine class, R ≤ 1 through
n = 512.** The container (previous section) was the outlier in
direction — desktop-class hardware is where faer's blocking looks
best, and even there it only reached parity.

## Standing consequences (recorded, not acted on — tuning freeze)

1. The kernels' "route the O(n³) bulk through faer's gemm" rule leaves
   ~15–20% on the table at L3 shapes ≤ 512 in the reference
   environment. A flat streaming gemm/trmm bulk path for the kernel
   layer is now a measured candidate lever — architect's call, next
   campaign material.
2. The common-thread thesis sharpens to its limit: on the target,
   blocked Level 3 is not merely a small win over streaming — in the
   shipping regime it is not a win at all. The two-speed machine is,
   on the reference class, one speed.
3. faer's 5–20× matmul wins over Pyodide are therefore entirely about
   scipy's OpenBLAS-generic build being far from the machine ceiling.
4. Unchanged rows stay unchanged for now: correctness machinery,
   size budget, and the freeze all say measure twice before rerouting.

Caveat kept open: all A/B builds are the plain simd128 profile; the
relaxed-SIMD (FMA) build could shift the balance toward faer's
microkernels and should be part of any rerouting campaign.
